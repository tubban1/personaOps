# 下一阶段任务规划：Ops Console / Run Dashboard

## 背景判断
`personaOps` 目前已经拥有可运行的核心执行内核，入口是 `main.py` 的 CLI，主链路由 `PipelineRunner` 串起 `Planning -> Media -> Publish -> Audit`。系统也已经在 `data/audit`、`data/assets` 等目录中持续沉淀运行结果。  
这意味着当前缺的不是“能不能跑”，而是“人能不能看懂、介入和复盘”。

## 为什么现在需要可视化层
1. 当前交互方式主要依赖 CLI 和日志，适合开发者，不适合运营、测试和协作验收。
2. 运行过程已经天然分成多个阶段，但缺少统一视图来展示一次任务的完整生命周期。
3. 审计、资产、文案、发布结果已经落盘，具备做可视化面板的数据基础。
4. 没有可视化层时，排查失败、核对产物、确认发布状态都要跨目录和跨日志手动拼接。

## 为什么不建议立刻做大而全前端
1. 当前核心模型和执行流程仍在演进，过早做重前端容易把不稳定的业务契约固化成 UI 依赖。
2. 现在最需要的是“可观察性”和“操作效率”，不是复杂的拖拽编排或多角色工作台。
3. 如果一开始就做完整产品形态，前端、接口、权限、状态同步都会把迭代成本放大。
4. 先把执行内核的可视化做好，更容易验证哪些交互是真需求，哪些只是想象中的需求。

## 建议目标形态
建议把下一阶段定位为一个轻量的 `Ops Console / Run Dashboard`，而不是传统意义上的业务官网或重型后台。

这个目标形态应该做到：
1. 一眼看到一次运行的状态、阶段、产物和错误。
2. 能从 run 进入 plan、media package、audit record 的详情页。
3. 能预览图片、文案和发布结果，而不是只看文件路径。
4. 能保留最少量的人工操作入口，例如重试、重新生成、重新发布。

## 分阶段任务拆解

### MVP：先做只读运行看板
目标是把现有执行链路“看得见”。

任务项：
1. 增加运行记录列表页，读取 `PipelineRunRecord`。
2. 增加运行详情页，展示 `post_plan_ids`、`media_package_ids`、`publish_audit_ids`。
3. 增加资产预览区，读取 `MediaPackage`、`RenderAudit`、`caption.json`、`render_log.json`。
4. 增加审计详情页，读取 `AuditRecord`。
5. 增加基础筛选：按模式、平台、账号、状态、时间过滤。

交付物：
1. 一个可访问的只读 Dashboard。
2. 一套稳定的读数据方式，优先基于本地文件和现有模型。
3. 一份页面信息结构说明，确保后续扩展不推翻重来。

依赖关系：
1. `app/runtime/models.py`
2. `app/media/models.py`
3. `app/audit/models.py`
4. `app/core/paths.py`
5. `data/audit` 和 `data/assets` 的现有落盘约定

风险：
1. 文件结构和字段命名变化会影响页面读取。
2. 某些历史任务可能缺少完整的审计或资产文件。
3. 只读看板如果没有统一数据层，容易变成“直接扫目录”的脆弱实现。

### 第二阶段：加入最小操作能力
目标是让看板不仅能看，还能做少量明确动作。

任务项：
1. 支持从 `PostPlan` 触发重新生成媒体资产。
2. 支持从 `MediaPackage` 触发重新发布。
3. 增加运行级别的重试和失败定位入口。
4. 增加任务状态时间线，串起 planning、media、publish、audit。
5. 增加操作确认和危险动作提示，尤其是真实发布相关入口。

交付物：
1. 一个可用于日常运维的轻量操作台。
2. 一组明确的动作 API 或 service 层函数。
3. 最少权限原则下的操作闭环。

依赖关系：
1. `app/runtime/pipeline_runner.py` 需要进一步拆分可复用步骤。
2. `app/runtime/manager.py` 需要更稳定的发布回执和失败信息。
3. `app/publish/session_manager.py` 和平台 driver 需要更清晰的会话边界。

风险：
1. 运行动作一旦开放，误操作成本会明显上升。
2. 发布链路涉及真实平台和账号，必须先区分 mock 与 real。
3. 如果业务层还没抽出独立 service，前端动作会直接绑死执行细节。

### 以后：再考虑完整协作平台
目标是把看板升级为真正的多人协作系统，但这一步应放在核心契约稳定之后。

可选方向：
1. 周计划编辑与审批流。
2. 多账号、多平台的统一调度。
3. 实时运行流和告警推送。
4. 角色权限、操作审计和回滚。
5. 更完整的内容资产管理与版本管理。

风险：
1. 这一步最容易把系统做成“看起来很完整，但很难维护”。
2. 如果基础数据模型不稳，协作层会频繁返工。

## 与当前代码结构的映射
当前可视化层最应该直接复用的模块如下：

1. `main.py`
   - 现有 CLI 入口，适合作为后续 UI 启动或本地调试入口的对照实现。
2. `app/runtime/pipeline_runner.py`
   - 现有主编排器，定义了一次运行的真实阶段边界。
3. `app/runtime/models.py`
   - `PipelineRunRecord` 是看板的主对象。
4. `app/media/models.py`
   - `MediaPackage` 和 `RenderAudit` 是资产预览与媒体阶段详情的核心数据源。
5. `app/runtime/manager.py`
   - `RuntimeManager` 是发布阶段和审计阶段的连接点。
6. `app/audit/models.py`
   - `AuditRecord` 是发布回执和最终结果的落点。
7. `app/planning/models.py`
   - `PostPlan`、`WeeklyPlan`、`ChannelProfile` 是规划页和过滤器的基础对象。
8. `app/core/paths.py`
   - 统一路径约定，适合为页面读取和索引服务提供基础。

## 推荐执行顺序
1. 先做只读 Dashboard，确认数据能被统一展示。
2. 再补最小操作能力，验证哪些动作是真正高频需求。
3. 最后再决定是否升级为完整协作平台。

## 本阶段的核心原则
1. 先可视化，再交互。
2. 先复用现有契约，再补新接口。
3. 先把运行事实看清楚，再谈复杂流程设计。
4. 任何涉及真实发布的能力，都要默认保守、可回退、可审计。
